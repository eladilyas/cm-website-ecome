# Internationalization

next-intl 4 with French default + English secondary. As-needed URL prefix.

## Routing rules

| URL | Locale | Notes |
|---|---|---|
| `/` | `fr` | Default — no prefix |
| `/en` | `en` | Explicit prefix for the non-default locale |
| `/admin/*` | n/a | Never localized — flat path, English-only |
| `/api/*` | n/a | Same — REST endpoints carry no UI language |

Implementation: [src/i18n/routing.ts](../src/i18n/routing.ts) declares `{ locales: ["fr","en"], defaultLocale: "fr", localePrefix: "as-needed" }`. The marketing tree lives under `src/app/[locale]/*`; everything else (admin, API) lives at the top level and the middleware bypasses next-intl for those paths.

Middleware logic (in [src/middleware.ts](../src/middleware.ts)):
```
if path starts with /admin or /api or /_next or /favicon → bypass next-intl
else                                                     → run next-intl middleware
```

## Message catalogs

```
messages/
├─ fr.json    primary — every key starts here
└─ en.json    translation — same shape as fr.json
```

Keys are nested by surface:

```jsonc
{
  "nav": {
    "menuLabels": { "store": "Magasin", ... },
    "solutionsDropdown": { ... }
  },
  "home": {
    "hero": { ... },
    "store": { ... }
  },
  "auth": {
    "signIn": { ... },
    "signUp": { ... },
    "fields": { ... },
    "validation": { ... }
  },
  "demo": { ... },
  ...
}
```

The structure mirrors UI surfaces, not features — so a translator can scan one section without context-switching.

## Adding a string

1. Add the key to `messages/fr.json` (canonical) with the French copy.
2. Add the same key to `messages/en.json` with the English copy.
3. Use it in a component:

```tsx
// Server component
import { getTranslations } from "next-intl/server";
const t = await getTranslations("nav.menuLabels");
return <span>{t("store")}</span>;

// Client component
"use client";
import { useTranslations } from "next-intl";
const t = useTranslations("nav.menuLabels");
return <span>{t("store")}</span>;
```

4. Run `npm run i18n:check` — it scans both files and fails on any key present in one but missing in the other.

## i18n parity script

`scripts/i18n-check.ts` walks both catalogs and reports:
- Keys in `fr.json` missing from `en.json` (and vice-versa)
- Empty values
- Type mismatches (a value that's an object in one but a string in the other)

CI runs this on every PR — see `.github/workflows/ci.yml`. Failures block merge.

## Locale-aware navigation

next-intl ships `Link`, `useRouter`, `usePathname`, `redirect` helpers in [src/i18n/navigation.ts](../src/i18n/navigation.ts). Always import these for **locale-prefixed routes** — using `next/link` would break the as-needed prefix logic.

Top-level routes that DON'T live under `[locale]` (admin, API) keep using vanilla `next/link` / `next/navigation`.

## Special cases

| Surface | How it handles i18n |
|---|---|
| Hard-coded React strings | Pulled into the catalog. The component imports a `t()` instance and references the key. |
| Plurals + interpolation | `t("orderCount", { count })` with `{count, plural, ...}` ICU syntax inside the catalog value. |
| Data-driven strings (product names, category labels) | Live in Postgres, not the catalog. They're returned in whatever language the seed/admin entered. Out of i18n scope. |
| Dates + numbers | `useFormatter()` from next-intl for runtime formatting (`formatter.dateTime`, `formatter.number`). |
| SEO metadata (title, description, canonical, hreflang) | Built by `generateMetadata()` in each page; reads from the catalog via `getTranslations()` server-side. |
| Demo simulator | Uses the same catalog but under a `demo.*` namespace so it's clearly isolated. |

## Bilingual surface tracker

Every public surface should ship in both languages. The list of surfaces that have been audited for parity lives implicitly in the catalog structure — if it's in `fr.json` AND `en.json` with the same shape, it's covered.

A handful of admin-side strings remain English-only by design (the operator panel isn't translated). They live inline in admin components rather than the catalog.

## Adding a third locale

If Arabic, Spanish, etc. is needed later:

1. Create `messages/<locale>.json` matching the fr.json shape.
2. Add the locale to `src/i18n/routing.ts` → `locales` array.
3. Update `next.config.ts` if any rewrites depend on the locale list.
4. Run `npm run i18n:check` — it now enforces parity across all three.
5. Audit the hreflang metadata generator in `src/app/sitemap.ts` and per-page `generateMetadata`.

Right-to-left scripts (Arabic, Hebrew) need additional work — Tailwind has `rtl:` variants, but a few hand-positioned UIs (the admin sidebar, the cart drawer) would need a layout pass.
